DIAGRAM 1: PATIENT LOGIN WITH JWT AUTHENTICATION (SD001)
================================================================================
Replaces your old "Admin Login" sequence diagram
Shows: Patient → Login Page → Auth Controller → Database → JWT Generation

@startuml
title Sequence Diagram: Patient Login with JWT Authentication (SD001)

actor Patient
participant "Login Page" as LoginPage
participant "Auth Controller" as AuthController
participant "Database" as DB
participant "JWT Service" as JWT

Patient -> LoginPage: Enter email and password
Patient -> LoginPage: Click "Login" button
LoginPage -> AuthController: login(email, password)
activate AuthController

AuthController -> AuthController: Validate email format
alt Email format invalid
    AuthController --> LoginPage: return error "Invalid email format"
    LoginPage --> Patient: Display error message
else Email format valid
    AuthController -> DB: getUserByEmail(email)
    activate DB
    
    alt User not found
        DB --> AuthController: return null
        deactivate DB
        AuthController --> LoginPage: return error "Account not found"
        LoginPage --> Patient: Display error message
    else User found
        DB --> AuthController: return user data
        deactivate DB
        
        AuthController -> AuthController: verifyPassword(password, hashedPassword)
        
        alt Password incorrect
            AuthController -> DB: incrementFailedAttempts(userId)
            activate DB
            DB --> AuthController: return attempt count
            deactivate DB
            
            alt Failed attempts >= 5
                AuthController -> DB: lockAccount(userId, 15 minutes)
                activate DB
                DB --> AuthController: return locked status
                deactivate DB
                AuthController --> LoginPage: return error "Account locked"
                LoginPage --> Patient: Display "Account locked for 15 minutes"
            else Failed attempts < 5
                AuthController --> LoginPage: return error "Invalid password"
                LoginPage --> Patient: Display error message
            end
        else Password correct
            AuthController -> DB: resetFailedAttempts(userId)
            activate DB
            DB --> AuthController: return success
            deactivate DB
            
            AuthController -> JWT: generateAccessToken(userId, role)
            activate JWT
            JWT --> AuthController: return accessToken (30 min expiry)
            deactivate JWT
            
            AuthController -> JWT: generateRefreshToken(userId)
            activate JWT
            JWT --> AuthController: return refreshToken (7 days expiry)
            deactivate JWT
            
            AuthController -> DB: logLoginEvent(userId, timestamp, ipAddress)
            activate DB
            DB --> AuthController: return success
            deactivate DB
            
            AuthController --> LoginPage: return {accessToken, refreshToken, userData}
            deactivate AuthController
            LoginPage -> LoginPage: Store tokens in localStorage
            LoginPage --> Patient: Redirect to Patient Dashboard
            Patient -> Patient: Successfully logged in
        end
    end
end

DIAGRAM 2: COMPLETE AI ANALYSIS WORKFLOW (SD002)
================================================================================
Shows: Patient → Upload → NSFW Filter → AI Model → Risk Assessment → Display
Most important - demonstrates core AI feature with external services

@startuml
title Sequence Diagram: Complete AI Analysis Workflow (SD002)

actor Patient
participant "Upload Page" as UploadPage
participant "Image Validator" as Validator
participant "NSFW Filter" as NSFWFilter
participant "Questionnaire" as Quiz
participant "AI Controller" as AIController
participant "Vision Transformer\n(Hugging Face)" as ViTModel
participant "Risk Assessor" as RiskAssessor
participant "Database" as DB
participant "Results Page" as ResultsPage

Patient -> UploadPage: Click "New Analysis"
UploadPage --> Patient: Display upload interface

Patient -> UploadPage: Select/capture skin lesion image
Patient -> UploadPage: Confirm upload
UploadPage -> Validator: validateImage(imageFile)
activate Validator

Validator -> Validator: Check file format (JPG/PNG)
Validator -> Validator: Check file size (<10MB)

alt Invalid file
    Validator --> UploadPage: return error "Invalid format or size"
    UploadPage --> Patient: Display error message
else Valid file
    Validator --> UploadPage: return success
    deactivate Validator
    
    UploadPage -> NSFWFilter: checkContent(imageData)
    activate NSFWFilter
    
    NSFWFilter -> NSFWFilter: Preprocess image (224x224, normalize)
    NSFWFilter -> NSFWFilter: Calculate skin percentage (4 criteria)
    
    alt Skin percentage < 25%
        NSFWFilter --> UploadPage: return error "Not a skin lesion"
        UploadPage --> Patient: Display "Upload actual skin photo"
    else Skin percentage >= 25%
        NSFWFilter -> NSFWFilter: Detect text/graphics
        
        alt Text/graphics detected
            NSFWFilter --> UploadPage: return error "Poster detected"
            UploadPage --> Patient: Display "Upload skin photo, not poster"
        else No text/graphics
            NSFWFilter -> NSFWFilter: Calculate NSFW score
            
            alt NSFW score > 0.35
                NSFWFilter --> UploadPage: return error "Inappropriate content"
                UploadPage --> Patient: Display error message
            else NSFW score <= 0.35
                NSFWFilter --> UploadPage: return validation passed
                deactivate NSFWFilter
                
                UploadPage -> Quiz: displayQuestionnaire()
                Quiz --> Patient: Display 8 symptom questions
                Patient -> Quiz: Answer all questions
                Patient -> Quiz: Submit questionnaire
                
                Quiz -> Quiz: Validate all required answers
                alt Incomplete questionnaire
                    Quiz --> Patient: Highlight missing questions
                else Complete questionnaire
                    Quiz -> DB: saveQuestionnaireResponses(patientId, responses)
                    activate DB
                    DB --> Quiz: return success
                    deactivate DB
                    
                    Quiz -> AIController: analyzeImage(imageData, symptoms)
                    activate AIController
                    AIController --> Patient: Display "Analyzing..." progress
                    
                    AIController -> ViTModel: loadModel("dima806/skin_cancer_detection_vit")
                    activate ViTModel
                    ViTModel --> AIController: return model loaded
                    
                    AIController -> ViTModel: predict(preprocessedImage)
                    ViTModel -> ViTModel: Perform neural network forward pass
                    ViTModel -> ViTModel: Generate probabilities for 7 classes
                    ViTModel --> AIController: return predictions {mel: 0.87, bcc: 0.05, ...}
                    deactivate ViTModel
                    
                    AIController -> AIController: Identify highest probability
                    AIController -> AIController: Convert to confidence percentage
                    AIController -> AIController: Perform lesion localization
                    
                    AIController -> RiskAssessor: assessRisk(cancerType, confidence, symptoms)
                    activate RiskAssessor
                    
                    alt Melanoma OR Confidence >= 85%
                        RiskAssessor -> RiskAssessor: Assign risk = URGENT, color = Red
                    else Basal Cell OR Actinic Keratoses
                        RiskAssessor -> RiskAssessor: Assign risk = HIGH, color = Orange
                    else Confidence 60-85% OR Concerning symptoms
                        RiskAssessor -> RiskAssessor: Assign risk = MEDIUM, color = Yellow
                    else Benign types with high confidence
                        RiskAssessor -> RiskAssessor: Assign risk = LOW, color = Green
                    end
                    
                    RiskAssessor -> RiskAssessor: Generate risk justification
                    RiskAssessor -> RiskAssessor: Calculate urgency score (0-100)
                    RiskAssessor --> AIController: return riskAssessment
                    deactivate RiskAssessor
                    
                    AIController -> DB: saveAnalysisReport(patientId, predictions, risk, image)
                    activate DB
                    DB --> AIController: return reportId
                    deactivate DB
                    
                    AIController --> ResultsPage: return analysisResults
                    deactivate AIController
                    
                    ResultsPage -> ResultsPage: Display risk level banner with color
                    ResultsPage -> ResultsPage: Show cancer type and confidence %
                    ResultsPage -> ResultsPage: Show all 7 classes with probabilities
                    ResultsPage -> ResultsPage: Display image with lesion bounding box
                    ResultsPage -> ResultsPage: Show symptom summary
                    ResultsPage -> ResultsPage: Provide recommended actions
                    ResultsPage -> ResultsPage: Display medical disclaimer (84% accuracy)
                    ResultsPage --> Patient: Show complete analysis results
                    Patient -> Patient: Review results and decide next action
                end
            end
        end
    end
end

================================================================================
Shows: Patient → Google Maps API → Hospital → Email System
Demonstrates multi-actor interaction with external services

@startuml
title Sequence Diagram: Book Appointment with Notifications (SD003)

actor Patient
participant "Hospital Finder" as HospitalFinder
participant "Google Maps API" as GoogleMaps
participant "Booking Page" as BookingPage
participant "Appointment Controller" as ApptController
participant "Database" as DB
participant "Email Service\n(SMTP)" as EmailService
participant "Hospital Staff" as Staff

Patient -> HospitalFinder: Click "Find Nearby Hospitals"
HospitalFinder -> Patient: Request location permission
Patient -> HospitalFinder: Grant GPS access
HospitalFinder -> HospitalFinder: Get current location (lat, lng)

HospitalFinder -> GoogleMaps: searchPlaces(location, radius=50km, type="hospital")
activate GoogleMaps
GoogleMaps -> GoogleMaps: Search within 50km radius
GoogleMaps -> GoogleMaps: Filter by "dermatology" OR "hospital"
GoogleMaps --> HospitalFinder: return hospitalList (up to 180 results)
deactivate GoogleMaps

HospitalFinder -> HospitalFinder: Sort by rating (4.5+ first) then distance
HospitalFinder --> Patient: Display map view and list view

Patient -> HospitalFinder: Select hospital from list
Patient -> HospitalFinder: Click "Book Appointment"

HospitalFinder -> BookingPage: openBookingForm(hospitalId)
BookingPage --> Patient: Display booking form

Patient -> BookingPage: Select analysis report from history
BookingPage -> DB: getAnalysisReport(reportId)
activate DB
DB --> BookingPage: return report {cancerType, riskLevel, date}
deactivate DB

BookingPage --> Patient: Display report summary
BookingPage --> Patient: Auto-suggest urgency based on risk

Patient -> BookingPage: Select preferred date
Patient -> BookingPage: Select preferred time slot
Patient -> BookingPage: Enter additional notes (optional)
Patient -> BookingPage: Check "Confirm info correct"
Patient -> BookingPage: Click "Submit Appointment Request"

BookingPage -> ApptController: createAppointment(patientId, hospitalId, reportId, preferences)
activate ApptController

ApptController -> ApptController: Validate all required fields
ApptController -> ApptController: Check date is not in past

alt Validation failed
    ApptController --> BookingPage: return error "Invalid data"
    BookingPage --> Patient: Display error message
else Validation passed
    ApptController -> DB: insertAppointment(appointmentData)
    activate DB
    DB -> DB: Create record with status = "Pending"
    DB -> DB: Link patient, hospital, and report
    DB --> ApptController: return appointmentId
    deactivate DB
    
    ApptController -> EmailService: sendStaffNotification(hospitalId, appointmentId)
    activate EmailService
    EmailService -> EmailService: Compose notification email
    EmailService -> EmailService: Include: Risk level, Cancer type, Preferred date
    EmailService -> Staff: Send email "New appointment request"
    EmailService --> ApptController: return emailSent = true
    deactivate EmailService
    
    ApptController --> BookingPage: return success {appointmentId, status: "Pending"}
    deactivate ApptController
    
    BookingPage --> Patient: Display "Request submitted successfully!"
    BookingPage --> Patient: Show "Hospital will confirm within 24-48 hours"
    BookingPage -> BookingPage: Redirect to "My Appointments" page
    BookingPage --> Patient: Display appointment status "Pending"
    
    note over Staff
        Hospital staff receives email
        and can review in dashboard
    end note
end

SYNTAX NOTES
================================================================================

PlantUML Sequence Diagram Syntax:
✅ actor "Name" as Alias - defines human actors
✅ participant "Name" as Alias - defines system components
✅ -> solid arrow (synchronous call)
✅ --> dashed arrow (return/response)
✅ activate/deactivate - shows active processing
✅ alt/else/end - conditional flows
✅ note over - adds explanatory notes
✅ note left/right - adds side notes

WHAT MAKES GOOD SEQUENCE DIAGRAMS?
================================================================================

GOOD PRACTICE:
✅ Clear actor/participant labels
✅ Activation bars show processing time
✅ Dashed arrows for returns
✅ Alt/else blocks for error handling
✅ Notes explain complex logic
✅ Left-to-right flow (chronological)
✅ Not too many participants (5-10 max)

AVOID:
❌ Too many nested alt blocks (max 2-3 levels)
❌ Crossing arrows (confusing)
❌ Too many participants (hard to read)
❌ Missing return arrows
❌ Unclear labels (use descriptive method names)

✅ Clear participant separation
✅ Proper activation/deactivation
✅ Error handling with alt blocks
