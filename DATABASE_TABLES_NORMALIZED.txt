================================================================================
DATABASE RELATIONSHIP SUMMARY
================================================================================

PRIMARY KEY RELATIONSHIPS:

1. profiles.user_id
   ├─> patient_data.user_id (1:1)
   ├─> doctors.user_id (1:1)
   ├─> medical_reports.patient_id (1:N)
   ├─> appointments.patient_id (1:N)
   └─> reviews.patient_id (1:N)

2. doctors.doctor_id
   ├─> appointments.doctor_id (1:N)
   └─> reviews.doctor_id (1:N)

3. medical_reports.report_id
   └─> appointments.report_id (1:N) - Optional

4. appointments.appointment_id
   └─> reviews.appointment_id (1:1) - Optional

5. audit_logs.user_id
   └─> profiles.user_id (Soft reference, no FK constraint)


FOREIGN KEY CONSTRAINTS:

- patient_data.user_id → profiles.user_id (ON DELETE CASCADE)
- doctors.user_id → profiles.user_id (ON DELETE CASCADE)
- medical_reports.patient_id → profiles.user_id (ON DELETE CASCADE)
- appointments.patient_id → profiles.user_id (ON DELETE CASCADE)
- appointments.doctor_id → doctors.doctor_id (ON DELETE CASCADE)
- appointments.report_id → medical_reports.report_id (ON DELETE SET NULL)
- reviews.patient_id → profiles.user_id (ON DELETE CASCADE)
- reviews.doctor_id → doctors.doctor_id (ON DELETE CASCADE)
- reviews.appointment_id → appointments.appointment_id (ON DELETE SET NULL)


NORMALIZATION NOTES:

1. Third Normal Form (3NF) achieved:
   - No repeating groups
   - All non-key attributes depend on primary key
   - No transitive dependencies

2. Single source of truth:
   - User information in profiles table only
   - Role-specific data in patient_data and doctors tables

3. Data integrity:
   - Foreign key constraints ensure referential integrity
   - CHECK constraints enforce business rules
   - UNIQUE constraints prevent duplicates

4. Flexible design:
   - JSONB for complex data (ai_prediction, metadata)
   - Optional relationships (report_id, appointment_id)
   - Soft references for audit logs
