5.4.2 White Box Testing

White box testing examines the internal logic of the system by tracing 
execution paths through the source code to verify that conditional branches, 
data handling, and error handling all behave correctly (Pirdaus et al., 2024). 
In this project, white box testing was applied to critical system components 
including the NSFW filter validation logic (25% skin threshold calculation and 
poster detection algorithms), the Vision Transformer AI preprocessing and 
inference pipeline, the risk assessment stratification logic (conditional 
branching for URGENT/HIGH/MEDIUM/LOW classification based on cancer type and 
confidence scores), the appointment workflow state machine (pending → confirmed/
rejected transitions with patient identity access control), database query 
functions with PostgreSQL transactions and foreign key relationships, the Google 
Maps API handler for hospital search with radius filtering and rating sorting, 
JWT token generation and validation middleware, password hashing with bcrypt, 
email notification service with SMTP error handling, and the privacy protection 
mechanism that conditionally reveals patient identity based on appointment 
status. Code coverage tools were used to ensure all conditional branches, loops, 
and exception handlers were exercised during testing.

5.4.2.1 NSFW Filter Validation Logic

The NSFW filter implements a multi-criteria validation algorithm that was tested 
through white box analysis to verify correct execution paths. The filter 
processes each image pixel to check four conditions: (1) RGB channel ordering 
where R > G > B indicating skin tones, (2) brightness range between 100-200 for 
realistic skin exposure, (3) saturation levels avoiding pure whites or blacks 
typical of graphics, and (4) overall skin-like pixel percentage calculation. 
The testing verified all conditional branches: when skin_percentage ≥ 0.25 the 
image passes validation and proceeds to AI analysis; when skin_percentage < 0.25 
the image is rejected with reason "Insufficient skin content"; when edge density 
exceeds threshold (indicating text/graphics) the image is rejected with reason 
"Poster detected"; when NSFW score > 0.35 the image is rejected with reason 
"Inappropriate content". Edge cases tested included boundary values (exactly 
0.25 skin percentage, exactly 0.35 NSFW score) to confirm inclusive/exclusive 
comparison operators, images with uniform colors to verify division-by-zero 
handling in saturation calculations, and completely black/white images to test 
brightness range validation. Code coverage analysis showed 100% branch coverage 
across all conditional statements in the NSFW filter module.

5.4.2.2 Risk Assessment Stratification Logic

The risk assessment module implements a decision tree with nested conditional 
statements to assign urgency levels based on AI predictions and symptom data. 
White box testing traced execution through all decision paths: (1) if cancer_type 
== "Melanoma" OR confidence ≥ 0.85, assign risk_level = "URGENT" and timeline = 
"Consult within 48 hours"; (2) elif cancer_type in ["Basal Cell Carcinoma", 
"Actinic Keratoses"], assign risk_level = "HIGH" and timeline = "Consult within 
1 week"; (3) elif confidence ≥ 0.60 AND confidence < 0.85, assign risk_level = 
"MEDIUM" and timeline = "Monitor and consult if changes"; (4) else assign 
risk_level = "LOW" and timeline = "Routine checkup recommended". Conservative 
bias logic was tested where borderline cases (confidence = 0.84) are rounded up 
to URGENT when combined with concerning symptoms (bleeding, rapid growth). The 
testing verified correct operator precedence in compound conditions (AND vs OR), 
boundary value handling at confidence thresholds (0.59, 0.60, 0.84, 0.85), and 
proper string matching for cancer type classification (case-insensitive matching, 
handling of whitespace). All conditional branches were executed and validated 
against expected risk level assignments. Edge cases included confidence scores 
at exact threshold values and cancer types with variations in spelling or 
capitalization to test string normalization logic.

5.4.2.3 Appointment Workflow State Machine

The appointment module implements a state machine with strict state transitions: 
Pending → Confirmed, Pending → Rejected, with no other transitions permitted. 
White box testing verified the conditional logic governing state changes and 
associated side effects. When staff confirms an appointment: (1) verify 
appointment.status == "Pending" (reject if not), (2) update appointment.status = 
"Confirmed", (3) execute patient identity reveal by updating staff access 
permissions, (4) insert audit log entry with staff_id and timestamp, (5) send 
confirmation email to patient, (6) add appointment to hospital calendar. When 
staff rejects an appointment: (1) verify appointment.status == "Pending", (2) 
update appointment.status = "Rejected", (3) ensure patient identity remains 
hidden (no permission update), (4) query Google Maps API for 3 alternative 
hospitals, (5) send rejection email with alternatives, (6) insert audit log 
entry indicating rejection without identity access. Testing verified all 
conditional checks execute correctly, database transactions commit atomically 
(all-or-nothing), and exception handling rolls back partial changes on failure. 
Race condition testing confirmed that concurrent confirmation attempts by 
different staff members are properly serialized using database row-level locking. 
Edge cases included attempting to confirm an already-confirmed appointment 
(should fail gracefully), attempting to access patient identity before 
confirmation (should return "Anonymous Patient"), and testing rollback behavior 
when email notification fails after status update.

5.4.2.4 Database Query Functions and Transactions

Critical database operations were tested for correct SQL query construction, 
parameter binding to prevent SQL injection, transaction handling, and foreign 
key constraint enforcement. White box testing examined: (1) Medical report 
insertion query with parameterized values for patient_id, image_url, ai_prediction 
(JSONB type), confidence (FLOAT), risk_level (ENUM), verifying foreign key 
reference to users.user_id is validated before insert; (2) Appointment query 
joining medical_reports and users tables with conditional WHERE clauses filtering 
by hospital_id and status, verifying correct JOIN syntax and NULL handling when 
report has no appointment; (3) Symptom response insertion with transaction 
wrapping to ensure report_id reference exists, testing rollback when parent 
report doesn't exist; (4) Audit log insertion with PostgreSQL RETURNING clause 
to retrieve generated log_id, testing concurrent inserts don't create race 
conditions. Testing verified all queries use parameterized statements 
(cursor.execute with %s placeholders) preventing SQL injection, foreign key 
violations trigger appropriate exceptions that are caught and logged, 
transactions use proper BEGIN/COMMIT/ROLLBACK blocks, and database connection 
pooling handles concurrent requests without deadlocks. Performance testing 
confirmed indexed columns (user_id, appointment_id, created_at) are used in 
WHERE clauses for efficient query execution.

5.4.2.5 JWT Token Generation and Validation

The authentication middleware implements JWT token creation and verification with 
security-critical logic tested through white box analysis. Token generation 
function: (1) creates payload with user_id, role, email, iat (issued at), exp 
(expiration = iat + 24 hours), (2) encodes using HS256 algorithm with SECRET_KEY, 
(3) returns token string. Token validation function: (1) decodes token using 
SECRET_KEY, (2) checks if current_time < exp (reject if expired), (3) verifies 
signature matches (reject if tampered), (4) extracts user_id from payload, (5) 
queries database to verify user still exists and is active, (6) returns user 
object if valid. White box testing verified: clock skew handling where tokens 
expiring within 60 seconds are renewed automatically, signature verification 
catches tampered tokens (modified user_id, extended expiration), expired token 
check uses correct comparison operator and timezone-aware timestamps, database 
lookup handles deleted users gracefully returning 401 Unauthorized, and exception 
handling catches PyJWT.DecodeError, PyJWT.ExpiredSignatureError separately with 
appropriate error responses. Edge cases tested included token issued in one 
timezone and validated in another (UTC normalization), token with exp exactly 
equal to current time (should fail), and malformed tokens with invalid base64 
encoding.

5.4.2.6 Password Hashing with Bcrypt

Password handling functions implement secure hashing using bcrypt with salt 
rounds = 12. White box testing examined: (1) hash_password function takes 
plain-text password, generates random salt, applies bcrypt with 12 rounds, 
returns hashed string; (2) verify_password function takes plain-text attempt and 
stored hash, applies bcrypt comparison, returns boolean; (3) password change 
function verifies current password before hashing new password. Testing verified: 
each hash generates unique salt (same password produces different hashes), 
verify_password correctly identifies matching and non-matching passwords, timing 
attacks are mitigated by bcrypt's constant-time comparison, and password strings 
are cleared from memory after hashing (using explicit del statement). Edge cases 
included empty password strings (rejected at validation layer before hashing), 
extremely long passwords (>72 bytes, bcrypt limit), passwords with Unicode 
characters (UTF-8 encoding before hashing), and null password values triggering 
appropriate exceptions.
