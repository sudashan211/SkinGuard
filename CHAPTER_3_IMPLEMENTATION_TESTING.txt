5.2.1 User Registration Page

The User Registration page enables new users to create accounts as either 
patients or hospital staff. The system collects essential information including 
full name, email address, password, phone number, and role selection. For 
hospital staff accounts, additional fields capture hospital name and medical 
license number for verification purposes. Upon form submission, the FastAPI 
backend validates the input data using Pydantic models, hashes the password 
using bcrypt with 12 salt rounds, generates a unique verification token, and 
stores the user record in the PostgreSQL database. A verification email 
containing a secure link is sent to the provided email address. Users must 
verify their email within 24 hours before gaining full access to the system. 
The implementation uses React controlled components with real-time validation 
feedback to ensure data integrity and guide users through the registration 
process.

5.2.2 Login Page

The Login page serves as the authentication entry point for both patients and 
hospital staff. Users enter their registered email and password credentials, 
which are transmitted securely to the FastAPI backend via HTTPS. The system 
retrieves the user record from the PostgreSQL database, verifies the password 
against the stored bcrypt hash, and checks the email verification status. Upon 
successful authentication, the backend generates a JWT (JSON Web Token) 
containing the user ID, role, and expiration timestamp (24 hours), which is 
returned to the React frontend and stored in localStorage. Subsequent API 
requests include this token in the Authorization header for session management. 
Failed login attempts are logged, and after three consecutive failures within 
15 minutes, the account is temporarily locked for security. The page is 
implemented using React form components with input validation and error message 
display.

5.2.3 Patient Dashboard Page

The Patient Dashboard page provides patients with a centralized overview of 
their skin health screening activities. Upon login, the system retrieves the 
user's medical reports from the PostgreSQL database and displays key metrics 
including total analyses performed, risk level distribution (Low, Medium, High, 
Urgent), pending appointments, and confirmed consultations. A timeline 
visualization shows the history of uploaded images with corresponding risk 
assessments, enabling patients to track changes over time. Quick action buttons 
provide direct access to core features: "New Analysis" (upload image), "Find 
Hospitals" (search nearby clinics), and "My Reports" (view history). The risk 
level distribution is rendered using Chart.js pie charts, while the appointment 
status is displayed using status badges with color coding. This dashboard serves 
as the primary navigation hub for patients and supports proactive skin health 
monitoring through visual trend indicators.

5.2.4 Image Upload and Capture Page

The Image Upload and Capture page implements the primary data collection 
interface for skin lesion analysis. Patients can either upload an existing image 
from their device or capture a new photo using their camera in real-time. The 
page provides clear visual guidelines showing optimal image characteristics: 
well-lit lesion, minimal background, close-up view, and supported formats 
(JPEG, PNG, maximum 10MB). Upon image selection or capture, a preview is 
displayed with options to retake or proceed. The selected image is transmitted 
to the FastAPI backend via multipart/form-data POST request, where it undergoes 
file format and size validation before being stored in the image storage system. 
The system generates a unique filename using UUID to prevent collisions and 
stores both the original image and a thumbnail (200×200 pixels) for performance 
optimization. The implementation uses React hooks (useState, useRef) to manage 
camera access through the browser's MediaDevices API, and a loading spinner 
indicates upload progress to enhance user experience.

5.2.5 Image Validation Page (NSFW Filter)

The Image Validation page implements critical content safety checks before 
proceeding to AI analysis. After image upload, the system automatically invokes 
the NSFW filter service which analyzes the image using four criteria: RGB 
channel ordering (checking if R>G>B pattern typical of skin tones), brightness 
range (verifying pixels fall within 100-200 range appropriate for skin), 
saturation levels (detecting oversaturated colors indicative of graphics), and 
overall skin-like pixel percentage. The system counts pixels meeting all four 
criteria and calculates the percentage against total image pixels. Images must 
achieve at least 25% skin-like content to pass validation. Additionally, edge 
density analysis detects text or graphic elements by computing Sobel gradients—
high edge density indicates posters or documents rather than organic skin 
texture. If validation fails, the page displays a specific rejection message 
("Poster detected - Please upload actual skin photo" or "Insufficient skin 
content - Ensure lesion is clearly visible") with visual feedback, and the image 
is not submitted for AI analysis. Validation results are logged to the 
image_validation table for quality monitoring and continuous improvement of 
threshold parameters.

5.2.6 Symptom Wizard Page

The Symptom Wizard page implements a structured four-step questionnaire to 
collect patient-reported symptoms that enhance AI analysis context. Step 1 
captures body location using an interactive body map diagram where patients 
click the affected area (face, neck, chest, back, arms, legs, hands, feet). 
Step 2 collects sensations through checkboxes: itching (none, mild, moderate, 
severe), pain (0-10 scale), burning sensation, and tenderness. Step 3 documents 
visual changes including color variations (darkening, lightening, multiple 
colors), size changes (growing, shrinking, stable), shape irregularity (smooth 
or jagged borders), and surface texture (flat, raised, scaly). Step 4 records 
temporal information: duration noticed (less than 1 month, 1-6 months, 6-12 
months, over 1 year), sun exposure history (minimal, moderate, extensive), and 
an optional text field for additional notes. Navigation buttons (Previous, Next) 
enable users to review and edit responses before final submission. The React 
implementation uses a state machine pattern to manage wizard progression and 
form validation, ensuring all required fields are completed. Upon completion, 
the structured symptom data is transmitted to the backend and stored in the 
symptom_responses table with foreign key reference to the medical report.

5.2.7 AI Analysis Results Page

The AI Analysis Results page displays comprehensive findings from the Vision 
Transformer model analysis. The page header features a prominent risk level 
banner with color coding: red for URGENT, orange for HIGH, yellow for MEDIUM, 
green for LOW. The primary prediction section shows the most likely cancer type 
(e.g., "Melanoma") with confidence percentage (e.g., "87%") and a brief 
description of the condition. A detailed predictions table lists all seven cancer 
types (Melanoma, Basal Cell Carcinoma, Squamous Cell Carcinoma, Actinic 
Keratoses, Benign Keratosis, Dermatofibroma, Vascular Lesion) with corresponding 
confidence scores, sorted from highest to lowest probability. The uploaded image 
is displayed with the AI-detected lesion area highlighted using a bounding box 
overlay. The risk assessment section provides justification text explaining why 
the particular risk level was assigned (e.g., "High confidence melanoma 
prediction requires urgent dermatologist evaluation"), recommended timeline 
(e.g., "Consult within 48 hours"), and next steps. A medical disclaimer 
prominently states: "This is a screening tool with 84% accuracy. Not a medical 
diagnosis. Consult a dermatologist for definitive evaluation." Action buttons 
enable patients to "Find Hospitals," "Download PDF Report," or "Save to 
History." The implementation uses React components with conditional rendering 
based on risk level and Chart.js for probability visualization.

5.2.8 Hospital Search Page

The Hospital Search page integrates Google Maps API to help patients locate 
nearby dermatology facilities. Upon page load, the system requests the patient's 
current location (with permission) or allows manual city/postal code entry. The 
FastAPI backend queries Google Places API with parameters: search radius 50km, 
place types "hospital" and "doctor," keyword "dermatology." Results include both 
specialized dermatology clinics and general hospitals with dermatology 
departments. The system retrieves up to 180 results and sorts them by rating 
(4.5+ stars prioritized) then distance. Each hospital card displays name, 
address, distance from patient, rating (stars), phone number, and an embedded 
Google Maps preview. Clicking "View Details" shows operating hours, patient 
reviews, and photos. The "Book Appointment" button initiates the appointment 
request workflow. Filter options enable patients to refine results by distance 
(5km, 10km, 25km, 50km), minimum rating (3+, 4+, 4.5+), and availability (open 
now, accepts new patients). The implementation uses React with Google Maps 
JavaScript API for interactive map display, marker clustering for performance, 
and Axios for backend API communication. The search results are cached in 
sessionStorage to avoid redundant API calls during the same session.

5.2.9 Appointment Booking Page

The Appointment Booking page enables patients to submit consultation requests 
to selected hospitals. After choosing a hospital from search results, patients 
access a booking form displaying the hospital name, address, and available 
consultation types (In-Person or Video Call). The form collects preferred date 
range (next 7 days, next 14 days, next 30 days), preferred time of day (morning, 
afternoon, evening), and an optional message to the hospital staff. Patients can 
attach their most recent AI analysis report or select from report history. Upon 
submission, the system creates an appointment record in the PostgreSQL database 
with status "Pending," links it to the medical report and patient ID, and stores 
the hospital information. The appointment appears in the hospital staff's pending 
queue as an anonymized report card—patient identity is not revealed at this 
stage to protect privacy. The patient receives an immediate confirmation message: 
"Request submitted. Hospital will review and respond within 48 hours." The 
appointment status can be tracked in the patient's dashboard under "My 
Appointments" section. The implementation uses React form components with date 
pickers (react-datepicker library) and file attachment handling.

5.2.10 Report History Page

The Report History page displays a chronological archive of all AI analyses 
performed by the patient. Upon page load, the system queries the medical_reports 
table filtered by the authenticated patient's user ID and renders report cards 
sorted by creation date (most recent first). Each card shows thumbnail of the 
uploaded image, analysis date, cancer type prediction, confidence score, risk 
level badge, and quick action buttons. Clicking "View Details" opens a modal 
displaying the complete analysis including all seven cancer type probabilities, 
symptom responses, risk justification, and recommended timeline. The "Download 
PDF" button generates a formatted medical report suitable for sharing with 
healthcare providers, containing patient information, image, AI predictions, 
risk assessment, and disclaimer. The "Book Follow-Up" button pre-fills the 
appointment booking form with this report attached. A comparison feature allows 
patients to select two reports from history and view them side-by-side to track 
lesion progression over time—particularly useful for monitoring changes. Filter 
and sort options enable organization by risk level, date range, or body location. 
The implementation uses React with pagination (15 reports per page), lazy loading 
for images, and jsPDF library for PDF generation.

5.2.11 Hospital Staff Dashboard Page

The Hospital Staff Dashboard page provides clinical personnel with a 
consolidated view of appointment workflows. Upon login with hospital staff 
credentials, the system retrieves appointments associated with the staff member's 
hospital from the PostgreSQL database and organizes them into three tabs: 
Pending Requests (anonymized reports awaiting review), Confirmed Appointments 
(scheduled consultations with revealed patient identities), and Past Appointments 
(completed or cancelled). The Pending tab displays report cards sorted by 
urgency (URGENT first, then HIGH, MEDIUM, LOW) showing AI analysis results, 
risk level, requested consultation type, and preferred date range—patient name, 
email, and phone number remain hidden. Each card includes "Review Report" and 
"Quick Reject" buttons. Statistics widgets show total pending requests, average 
response time, and urgency distribution. The Confirmed tab displays a calendar 
view with scheduled appointments, patient names (now visible), consultation 
types, and preparation notes. The Past tab maintains historical records for 
audit purposes. A notification badge alerts staff to new urgent cases requiring 
immediate attention. The implementation uses React with FullCalendar library for 
appointment visualization, real-time updates via WebSocket for new request 
notifications, and role-based access control ensuring only authorized staff can 
access their hospital's data.

5.2.12 Report Review Page (Hospital Staff)

The Report Review page enables hospital staff to thoroughly examine AI-generated 
skin lesion analysis before confirming appointments. Accessible from the pending 
queue, the page displays the complete medical report including high-resolution 
lesion image with zoom capability, all seven cancer type predictions with 
confidence scores, detailed symptom responses from the patient's questionnaire 
(body location, sensations, visual changes, duration), risk assessment 
justification, and AI model metadata (ViT-H-14, accuracy 84%). A privacy notice 
at the top states: "Patient identity hidden until appointment confirmed." Staff 
can evaluate the urgency by reviewing the risk level banner, examine the lesion 
characteristics against the symptom data for consistency, and assess whether the 
case requires in-person or video consultation. Action buttons at the bottom 
provide three options: "Confirm Appointment" (proceeds to scheduling interface), 
"Reject with Alternatives" (generates list of 3 nearby hospitals for patient 
referral), or "Request More Information" (sends anonymized message to patient 
asking clarifying questions). An annotation feature allows staff to mark areas 
of concern on the image for consultation preparation. The implementation uses 
React with image zoom library (react-image-magnifiers), and all staff actions 
are logged to the audit_logs table with timestamps for HIPAA compliance.

5.2.13 Appointment Confirmation Page (Hospital Staff)

The Appointment Confirmation page implements the scheduling workflow after staff 
decide to accept a patient's request. The page displays the patient report 
summary at the top and a scheduling form below. Staff select an available date 
and time using an interactive calendar that shows existing appointments (to 
prevent double-booking), choose consultation type (In-Person or Video Call—if 
video, system generates unique meeting room URL), estimate consultation duration 
(15, 30, 45, or 60 minutes), and optionally enter special instructions for the 
patient (e.g., "Please bring previous medical records" or "Fasting not 
required"). Upon clicking "Confirm and Notify Patient," the system performs 
multiple actions: updates appointment status to "Confirmed" in the database, 
reveals patient identity to staff by unlocking access to the users table record 
(name, email, phone, date of birth, medical history), adds the appointment to 
the hospital calendar, sends confirmation email to the patient containing 
appointment details and video link (if applicable), logs the identity access 
event to audit_logs with staff ID and timestamp, and moves the appointment from 
Pending to Confirmed tab. A success message displays: "Appointment confirmed. 
Patient notified via email." Staff can now view the patient's full profile 
including previous reports and medical history for consultation preparation. The 
implementation uses React form components with date-time picker 
(react-datetime-picker), integration with hospital calendar systems via iCal 
format, and email sending via SMTP service.

5.2.14 Appointment Rejection Page (Hospital Staff)

The Appointment Rejection page handles cases where staff cannot accommodate a 
patient's request. Upon clicking "Reject Request" from the report review page, 
staff access a rejection form requiring a reason selection from predefined 
options: "No availability in next 30 days," "Requires specialized dermatology 
center," "Case severity beyond our capacity," or "Other" (with text field). The 
system automatically queries Google Maps API to find three alternative hospitals 
within 50km that match the patient's needs, considering the case risk level 
(URGENT cases recommended to university hospitals, HIGH/MEDIUM to dermatology 
clinics). Staff can review the suggested alternatives and optionally add a 
personalized message (e.g., "We recommend Dr. Lee's clinic which specializes in 
melanoma cases"). Upon confirmation, the system updates appointment status to 
"Rejected," sends a rejection email to the patient containing the reason, 
alternative hospital recommendations with contact details and Google Maps links, 
and encouraging message to seek care elsewhere. Critically, patient identity is 
never revealed to staff during rejections—the name, email, and phone number 
remain encrypted and inaccessible. This privacy protection mechanism ensures 
rejected patients maintain anonymity. The rejection is logged to audit_logs 
without identity access. The implementation uses React with multi-step form 
validation and email template rendering for professional communication.

5.2.15 Patient Medical History Page (Hospital Staff)

The Patient Medical History page provides hospital staff with comprehensive 
longitudinal health data after appointment confirmation. Accessible only for 
confirmed appointments (identity revealed), the page displays the patient's 
complete profile including personal information (name, date of birth, contact 
details), all previous medical reports from the SkinGuard system sorted 
chronologically, and uploaded medical documents (if any). A timeline 
visualization shows the progression of skin lesion analyses, enabling staff to 
identify patterns such as recurring issues on the same body location or 
increasing risk levels over time. Each historical report can be expanded to view 
full details including images, AI predictions, symptoms, and risk assessments. 
The comparison feature allows staff to select two or more reports and display 
them side-by-side for clinical evaluation of lesion changes. Notes section 
enables staff to record consultation observations, diagnosis, treatment plan, 
and follow-up recommendations which are stored in the consultation_notes table. 
All access to this page is logged to audit_logs with timestamp and staff ID for 
HIPAA compliance. The implementation uses React with timeline component 
(react-chrono library), PDF viewer for medical documents, and image comparison 
slider for before/after analysis.

5.2.16 Profile Management Page

The Profile Management page enables all authenticated users (patients and 
hospital staff) to view and update their account information. Upon navigation 
to the profile page, the system retrieves the user's record from the PostgreSQL 
database based on the JWT token and displays current details including full 
name, email address, phone number, date of birth (patients), and hospital 
affiliation (staff). An edit mode allows users to modify updatable fields—email 
changes require re-verification, phone number updates are instant. Password 
change functionality requires entering the current password for security, then 
new password (minimum 8 characters, must include uppercase, lowercase, number, 
special character) with confirmation. Hospital staff can update their medical 
license number if it changes. Profile photo upload enables users to add a 
picture (max 2MB, square crop tool provided) which is stored in image storage 
and displayed across the application. Account deletion option is available with 
strong confirmation ("Type DELETE to confirm") and explanation of consequences 
(all reports and appointments deleted, irreversible). Two-factor authentication 
(2FA) toggle allows security-conscious users to enable SMS or email-based second 
factor. All profile updates trigger email notifications to alert users of changes 
for security. The implementation uses React form components with real-time 
validation, image cropping library (react-image-crop), and secure password 
update API endpoint.

5.4 Testing

behaves as intended and meets user expectations before deployment. In this

patients performing skin lesion analyses and hospital staff reviewing reports 
and confirming appointments. Together, these testing activities verify both the 
technical correctness of individual components and the overall usability, 
accuracy, and clinical relevance of the AI-powered screening system.

5.4.2 Integration Testing

Integration testing validates the interaction between different system 
components to ensure they work together correctly. Key integration test scenarios 
included: frontend-backend API communication verifying that React axios requests 
receive correct JSON responses from FastAPI endpoints with proper status codes 
(200 for success, 400 for validation errors, 401 for unauthorized, 500 for server 
errors); Vision Transformer AI model integration confirming that image 
preprocessing, model inference, and prediction parsing work end-to-end with 
response times under 10 seconds; NSFW filter integration ensuring validation 
results correctly block or pass images before AI analysis; Google Maps API 
integration verifying hospital search queries return results within 3 seconds 
with proper distance calculations and rating sorting; PostgreSQL database 
transactions testing that foreign key relationships maintain referential integrity 
(e.g., medical reports reference valid patient IDs, appointments reference valid 
report IDs); email notification system confirming SMTP connection, template 
rendering, and successful delivery for registration, appointment confirmation, 
and rejection scenarios; authentication flow validating that JWT tokens are 
correctly generated on login, included in subsequent requests, validated by 
middleware, and rejected when expired; and file upload integration ensuring 
images are stored in the correct directory structure with unique filenames and 
accessible URLs. All integration tests passed successfully with no critical issues, 
demonstrating robust inter-component communication.

5.4.3 User Acceptance Testing (UAT)

User Acceptance Testing evaluates the system from the end-user perspective to 
validate usability, clinical relevance, and real-world applicability. UAT was 
conducted with 8 participants: 5 potential patients (ages 25-55, mixed 
technology proficiency) and 3 healthcare professionals (2 dermatology nurses, 
1 general practitioner). Patient participants completed the full workflow: 
registration, image upload of sample skin lesions (provided by researchers to 
ensure consistency), symptom questionnaire completion, AI results review, 
hospital search, and appointment booking. Healthcare participants tested the 
hospital staff dashboard: reviewing anonymized reports, confirming and rejecting 
appointments, and accessing patient medical history post-confirmation. Evaluation 
criteria included ease of use (5-point Likert scale), task completion rate, time 
to complete workflows, error rate, and satisfaction. Results showed 92% task 
completion rate (46/50 tasks completed without assistance), average satisfaction 
rating of 4.3/5.0, and average time of 3.5 minutes for complete patient analysis 
workflow. Key feedback: patients appreciated the clear risk level visualization 
and hospital recommendations, but suggested adding progress indicators during AI 
analysis; healthcare professionals valued the urgency-based sorting of pending 
appointments but requested bulk confirmation features for efficiency. Minor 
usability issues were identified (unclear body location selector labels, 
insufficient contrast on some buttons) and subsequently fixed. Overall, UAT 
confirmed the system meets user expectations and is ready for pilot deployment.
