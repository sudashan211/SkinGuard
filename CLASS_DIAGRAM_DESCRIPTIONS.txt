CLASS DESCRIPTIONS
================================================================================

The User class stores core authentication credentials and personal data, identified by a primary key 'id' (UUID), along with 'email', 'password_hash', and 'full_name'. It includes a 'role' field to differentiate between patients, hospital staff, and administrators, and a 'verified' boolean to track email verification status. Additional attributes like 'avatar_url' and 'language_preference' personalize the user experience. Functions 'register()', 'login(email, password)', and 'logout()' handle authentication workflows, while 'updateProfile()' and 'resetPassword(email)' enable profile management. The 'verifyEmail(token)' function ensures secure account activation, protecting against unauthorized registrations.

The PatientProfile class extends the User entity with health-specific data, linked via 'user_id' foreign key. It stores demographic information like 'age', 'gender', and 'skin_type' (Fitzpatrick scale I-VI), along with comprehensive medical history including 'medical_conditions', 'current_medications', 'allergies', 'previous_skin_conditions', 'family_history', and 'sun_exposure_level'. Functions 'createProfile()' and 'updateProfile()' manage patient data entry, while 'getHealthHistory()' retrieves complete medical records. The 'calculateRiskFactors()' function analyzes patient attributes to identify individuals at higher risk for skin cancer, enabling personalized risk assessments and tailored AI predictions.

The HospitalStaff class stores credentials and facility information for medical professionals, identified by 'id' and linked to 'user_id'. It includes verification credentials like 'license_no' (unique medical license number) and professional details such as 'hospital_name', 'specialization', 'bio', 'education', and 'certifications'. Geographic coordinates ('latitude', 'longitude') enable location-based hospital searches, while 'clinic_hours' and 'languages' assist patients in finding suitable providers. The 'average_rating' and 'review_count' fields track service quality. Functions 'updateFacilityInfo()' maintains current hospital data, 'viewPendingReports()' displays unconfirmed appointments, 'confirmAppointment(id)' and 'rejectAppointment(id, reason)' manage booking workflows, and 'accessPatientHistory(id)' retrieves relevant medical reports with privacy protection until confirmation.

The AnalysisReport class stores AI-generated skin lesion analysis results, identified by 'report_id' and linked to 'patient_id'. It preserves both original ('image_url') and compressed ('image_thumbnail') images for performance optimization. AI predictions are stored in 'cancer_type' (primary classification), 'confidence_score' (prediction probability), 'all_predictions' (JSON array of all seven cancer types with probabilities), and 'risk_level' (low, medium, high, urgent). The 'risk_justification' field provides explainable AI output, while 'lesion_location', 'body_location', and 'symptom_responses' (JSON from symptom wizard) contribute to comprehensive analysis. The 'model_version' field tracks which AI model version generated the prediction, ensuring reproducibility. Functions 'saveReport()' persists analysis to database, 'generatePDF()' creates downloadable reports, 'shareWithDoctor(email)' enables secure sharing, and 'getReportSummary()' formats data for patient dashboards.

The SymptomQuestionnaire class captures patient-reported symptoms through a structured four-step wizard, linked to 'report_id'. It stores temporal information ('duration_noticed'), observable changes ('size_changes'), sensory symptoms ('pain_level', 'itching_level'), alarming signs ('bleeding_frequency'), anatomical context ('lesion_location'), environmental factors ('sun_exposure'), and freeform observations ('additional_notes'). All responses are collected before image upload, ensuring comprehensive clinical context for AI analysis. Functions 'validateResponses()' ensures data completeness and format correctness, 'calculateSymptomScore()' quantifies symptom severity for risk stratification, and 'getRedFlags()' identifies urgent warning signs (rapid growth, bleeding, pain) that automatically elevate risk level to 'urgent', overriding AI confidence scores to ensure conservative medical caution.

The ImageValidation class enforces content safety and quality standards before AI analysis, storing validation metrics for each uploaded image. It tracks 'skin_percentage' (minimum 25% required), 'nsfw_score' (explicit content detection), and 'text_detected' (poster/document rejection) to ensure images depict genuine skin lesions. The 'validation_passed' boolean and 'rejection_reason' text provide clear feedback for failed uploads. Functions 'validateSkinContent()' calculates skin-like pixel percentage using RGB color analysis, 'detectTextOrGraphics()' identifies non-medical images through edge density and contrast patterns, 'checkNSFW()' screens inappropriate content, and 'logValidationResult()' creates audit trails for debugging and system improvement. This class prevents model misuse and ensures data quality for accurate AI predictions.

The AIModel class configures and manages the Vision Transformer AI system, storing model metadata like 'model_id', 'name' (e.g., "ViT-H-14"), 'version', 'provider' (Hugging Face), and 'model_path' (storage location). It tracks performance metrics ('accuracy' of 84% on test dataset) and supported 'cancer_types' (JSON array of seven classifications: melanoma, basal cell carcinoma, squamous cell carcinoma, actinic keratosis, benign keratosis, dermatofibroma, vascular lesion). The 'active' boolean enables model switching for A/B testing or upgrades. Functions 'loadModel()' initializes the AI pipeline, 'predict(image)' generates cancer type probabilities, 'preprocessImage(image)' normalizes input dimensions and pixel values, and 'postprocessResults(output)' formats raw model output into structured JSON predictions with confidence scores.

The RiskAssessment class translates AI predictions into actionable medical guidance, linking 'report_id' to stratified risk levels. It stores 'cancer_type' (highest probability classification), 'confidence' (model certainty), 'risk_level' (low, medium, high, urgent), 'urgency_score' (0-100 numerical scale), 'timeline' (recommended consultation timeframe), and 'recommended_action' (patient-friendly advice). Functions 'calculateRiskLevel()' applies thresholds (low <40%, medium 40-70%, high 70-85%, urgent >85% or symptom red flags), 'generateJustification()' creates human-readable explanations citing confidence scores and symptom data, 'applyConservativeBias()' intentionally elevates borderline cases to favor false positives over false negatives (ethical medical AI), and 'assignUrgencyScore()' quantifies urgency for appointment prioritization and notification triggers.

The Appointment class manages the complete patient-hospital booking workflow, identified by 'appointment_id' and linking 'patient_id', 'hospital_id', 'report_id' (optional analysis report attachment), with both patient-requested times ('preferred_date', 'preferred_time') and staff-confirmed times ('confirmed_date', 'confirmed_time'). It stores consultation parameters like 'consultation_type' (in-person or video), 'duration' (minutes), bilateral communication ('patient_notes', 'staff_notes'), workflow 'status' (pending, confirmed, completed, cancelled), and 'urgency_level' derived from risk assessment. Functions 'createRequest()' initiates booking, 'confirmByStaff(details)' approves and schedules appointments, 'rejectByStaff(reason)' declines with explanation, 'cancel()' allows mutual cancellation, 'sendReminder()' triggers email notifications before scheduled time, and 'updateStatus(status)' tracks appointment lifecycle for analytics and patient dashboards.

The Hospital class aggregates dermatology facilities from Google Maps API, storing both Google data ('google_place_id', 'name', 'address', 'phone', 'email', 'website') and enriched metadata ('latitude', 'longitude', 'rating', 'review_count', 'operating_hours', 'specialties' as JSON arrays). The 'distance_km' field stores calculated distance from patient location, and 'is_open' indicates current availability based on operating hours and local time. Functions 'searchNearby(lat, lng, radius)' queries Google Maps within 50km radius, 'filterByRating(min)' prioritizes highly-rated facilities (4.5+ stars preferred), 'sortByDistance()' ranks results spatially, 'getDirections(origin)' generates Google Maps navigation URLs, and 'checkAvailability()' validates if the facility accepts appointments. This class implements the two-tier hospital search strategy: first show dermatology specialists, then general hospitals if <20 results.

The EmailNotification class manages asynchronous email delivery for all system communications, storing 'recipient_email', 'recipient_name', 'subject', 'body' (HTML content), template identifiers ('template_name', 'template_data' as JSON for dynamic content), delivery 'status' (pending, sent, failed), timestamps ('sent_at', 'created_at'), and 'error_message' for debugging. Functions 'sendConfirmationEmail()' notifies patients of confirmed appointments with calendar attachments, 'sendRejectionEmail()' explains staff decisions and suggests alternatives, 'sendPasswordResetEmail()' delivers secure reset tokens, 'sendReminderEmail()' prevents no-shows with 24-hour advance alerts, and 'retryFailedEmail()' implements exponential backoff for transient delivery failures. This class abstracts SMTP complexity and enables centralized email logging for compliance and user communication auditing.

The AuditLog class provides comprehensive security and compliance logging, recording every sensitive system action with 'user_id', 'action' description (login, upload, analyze, appt_book, privacy_access), 'resource_type' (user, report, appointment), 'resource_id' (affected entity UUID), network metadata ('ip_address', 'user_agent'), and structured 'metadata' (JSON) for context-specific details like failed login attempts or privacy policy changes. Functions 'logAction(action, resource)' creates timestamped audit entries, 'logPrivacyAccess()' specifically tracks when hospital staff view patient data (GDPR compliance), 'logLoginAttempt()' monitors authentication for intrusion detection, and 'generateAuditReport()' compiles forensic-ready activity reports for security reviews. This class enables regulatory compliance (HIPAA, GDPR), security incident investigation, and data breach notifications.

The GoogleMapsAPI class encapsulates external mapping service integration, configured with 'api_key' (user's personal Google Maps API key), 'search_radius' (default 50km), and 'max_results' (default 180) parameters. Functions 'searchPlaces(location, type)' performs Places API queries filtered by keywords ("dermatology", "skin specialist", "hospital"), 'getPlaceDetails(placeId)' retrieves comprehensive facility information (hours, photos, reviews, contact), 'getDirections(origin, dest)' generates navigation routes with distance and duration, and 'geocodeAddress(address)' converts text addresses to coordinates. This class abstracts Google Maps API complexity, handles rate limiting (2500 free requests daily), manages error responses (invalid API key, quota exceeded), and implements caching to minimize billable API calls while maintaining fresh hospital data.

The VisionTransformerAI class interfaces with the Hugging Face Vision Transformer model, configured with 'model_name' ("ViT-H-14 fine-tuned"), 'provider' (Hugging Face Hub), 'api_endpoint' (model inference URL), and documented 'accuracy' (84% on HAM10000 test set). Functions 'loadModel()' initializes the transformer architecture and loads pre-trained weights, 'predict(image)' runs inference to generate probability distributions across seven cancer types, 'classifyLesion(image)' returns top prediction with confidence score, and 'localizeLesion(image)' applies attention mapping for explainable AI visualization (future enhancement). This class handles model versioning, input preprocessing (resize to 224x224, normalize RGB channels), batch processing for efficiency, and GPU acceleration when available, abstracting deep learning complexity from application business logic.

The SMTPService class manages outbound email delivery infrastructure, configured with 'smtp_host' (mail server address), 'smtp_port' (typically 587 for TLS or 465 for SSL), 'username' (authenticated sender account), and 'use_tls' (encryption flag). Functions 'sendEmail(to, subject, body)' delivers individual messages with HTML formatting and attachment support, 'sendBulkEmails(recipients)' efficiently dispatches mass notifications (appointment reminders, system updates), and 'validateEmailAddress(email)' performs syntax and domain validation before attempting delivery. This class implements connection pooling for performance, automatic retry with exponential backoff for transient failures (network timeouts, temporary mailbox unavailability), error categorization (permanent vs transient failures), and bounce handling to maintain sender reputation and prevent blacklisting.

RELATIONSHIP DESCRIPTIONS (OPTIONAL - FOR DETAILED DOCUMENTATION)
================================================================================

If your thesis requires detailed relationship explanations, include these:

The User class has a one-to-zero-or-one relationship with PatientProfile and HospitalStaff, implementing role-based access control where each user account extends into exactly one role-specific profile. This design enforces the principle of least privilege and data separation, ensuring patients cannot access hospital staff functions and vice versa.

The User class has a one-to-many relationship with AnalysisReport, allowing patients to maintain comprehensive medical history by uploading multiple skin lesion images over time. Each report is immutably linked to its creator, enabling longitudinal tracking of lesion changes, treatment effectiveness monitoring, and data ownership enforcement for privacy compliance.

The AnalysisReport class has mandatory one-to-one composition relationships with SymptomQuestionnaire, ImageValidation, and RiskAssessment, ensuring every analysis includes validated image content, patient-reported symptoms, and risk stratification. This composition pattern guarantees referential integrity and prevents orphaned data, as deletion of an analysis report automatically cascades to remove all associated components.

The Appointment class has many-to-one associations with Hospital and HospitalStaff, modeling the real-world scenario where multiple patients book appointments at the same facility and with the same staff members. The optional link to AnalysisReport (zero-or-one relationship) allows both report-driven consultations and general dermatology appointments without pre-existing AI analysis.

External service classes (VisionTransformerAI, GoogleMapsAPI, SMTPService) use dependency relationships (dashed arrows) rather than direct associations, indicating they are external systems accessed via APIs rather than database entities. These dependencies highlight third-party integrations that require API keys, network connectivity, and error handling for service unavailability.
